### 1. Root Cause Hypothesis
The root cause is the cancellation of the `self._socket.recv()` coroutine by `asyncio.wait_for()` during timeouts. Repeated cancellations corrupt the ZeroMQ socket's internal state machine (specifically the REP socket's strict request-reply sequence), causing it to stop processing new messages after the underlying TCP connection is closed by the peer (likely due to a 50-second system-level TCP keepalive timeout).

### 2. Problematic Line(s)
```python
raw = await asyncio.wait_for(self._socket.recv(), timeout=1.0)  # Cancels recv() on timeout
```

### 3. Symptom Mechanism
- During idle periods, timeouts trigger every 1 second, repeatedly canceling `recv()`.
- After ~50 seconds (typical system TCP keepalive timeout), the client's OS closes the connection.
- The cancelled `recv()` leaves the REP socket in an inconsistent state where it ignores the dead connection, causing subsequent `recv()` calls to hang indefinitely instead of detecting the closed connection or timing out.

### 4. Suggested Fix
Replace the `wait_for` approach with native ZeroMQ timeouts using socket options:

```python
async def _serve_loop(self) -> None:
    # Set socket timeout once (outside loop)
    self._socket.setsockopt(zmq.RCVTIMEO, 1000)  # 1000ms timeout

    while self._running:
        try:
            raw = await self._socket.recv()  # Uses ZeroMQ's internal timeout
        except zmq.Again:  # Raised on native timeout
            continue
        except asyncio.CancelledError:
            raise
        except Exception:
            logger.exception("Ошибка приёма команды ZMQ")
            continue
        
        # ... rest of handler code unchanged ...
```

**Key changes:**
1. Set `RCVTIMEO` socket option to handle timeouts natively
2. Replace `wait_for` with direct `await self._socket.recv()`
3. Catch `zmq.Again` for timeouts (no cancellation occurs)
