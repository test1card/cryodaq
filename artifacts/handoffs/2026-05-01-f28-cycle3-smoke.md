### Root cause of latency

`gemma4:e4b` is a **thinking-first model**:
1. Generates ~570 internal reasoning tokens (invisible in `response` field)
2. Then generates ~150 visible response tokens
3. Total budget needed: 2048+ (was 300 → empty response; now fixed to 2048)
4. Context window: **131072 tokens** (vs expected ~8K for e4b model)
   — huge KV cache allocation likely contributing to slow prefill

## Architect decision needed: latency mitigation

| Option | Estimated latency | Notes |
|---|---|---|
| **A. Accept ~50s** | ~50s | Async/non-blocking; spec allows "Гемма думает..." state; operator gets insight eventually |
| **B. num_ctx=4096** | untested (Ollama busy) | Add to agent.yaml: uncomment `# num_ctx: 4096` |
| **C. Switch to qwen3:14b** | ~25s (Cycle 1 smoke) | Known fast; tested; change `default_model` in agent.yaml |
| **D. Wait: test B explicitly** | unknown | Schedule separate test session |

**My assessment:**
- Option A: Acceptable from UX perspective — spec says non-blocking, operator sees "Гемма думает...". Safety is not affected.
- Option B: High potential for speedup (reducing 128K context → 4K should dramatically cut prefill time). Low risk. Recommend testing.
- Option C: qwen3:14b works but is 9.3GB — may not load simultaneously with gemma4:e4b (M5 24GB total); also loses Gemma model identity.

**Recommendation:** Test Option B first (uncomment `num_ctx: 4096` in agent.yaml). If prefill drops from 22s to <3s with 4K context, total latency would be ~10-15s.
