# Руководство оператора CryoDAQ

Версия документа: Montana candidate, 17 июля 2026. Релизная основа — v0.64.1;
кандидат ещё требует гейтов из `PROJECT_STATUS.md` и
`docs/lab_verification_checklist.md`.

## 1. Назначение

CryoDAQ управляет сбором данных и операторскими действиями на криогенном стенде. Система:

- читает температурные каналы LakeShore 218S
- читает и управляет Keithley 2604B по каналам `smua` и `smub`
- читает давление, если прибор настроен
- пишет данные в SQLite
- отслеживает алармы и safety state
- хранит операторские записи и метаданные экспериментов
- формирует отчёты и ведёт архив экспериментов

Актуальная продуктовая модель:

- один эксперимент = одна experiment card
- во время активного эксперимента есть ровно одна открытая карточка
- по завершении карточка закрывается и становится архивной записью
- следующий эксперимент создаёт новую карточку
- операторский журнал является частью карточки эксперимента, а не её заменой

Важно: `engine` и GUI разделены. Закрытие GUI не останавливает сбор данных.

## 2. Порядок запуска

Рекомендуемый полный операторский запуск — `cryodaq` (на Windows также через
ярлык). Launcher process содержит Qt GUI и владеет lifecycle отдельных engine,
bridge и optional assistant. Bounded report children принадлежат engine или
assistant-компоненту, который их запросил, а не launcher напрямую. GUI не
является отдельным child этого launcher, но остаётся клиентом backend truth и
не получает instrument authority.

Сокращённый диагностический запуск:

1. Запустите `cryodaq-engine`.
2. После запуска engine откройте отдельный `cryodaq-gui`.

Этот путь намеренно не поднимает launcher-owned assistant/periodic-delivery
supervision; on-demand reporting самого engine остаётся доступным.
Дополнительно доступны:

- `start.sh` / `start.bat` — полный launcher одной командой
- `start_mock.sh` / `start_mock.bat` — то же в режиме эмуляции (`CRYODAQ_MOCK=1`)
- `cryodaq-engine --mock` — engine без реальных приборов
- web dashboard через `uvicorn cryodaq.web.server:app --host 127.0.0.1 --port 8080`
  (доступен также из ToolRail: «Ещё» → «Открыть Web-панель»)

Web dashboard: GET-эндпоинты только для чтения и работают по loopback-модели
доверия без токена — биндите только `127.0.0.1`, для доступа из сети используйте
SSH-туннель. Ровно два write-эндпоинта (`POST /api/v1/log` — запись в журнал,
`POST /api/v1/alarms/{id}/ack` — подтверждение тревоги) требуют Bearer-токен из
gitignored `config/web.local.yaml` (`web.api_token`); пока токен не задан, эти
эндпоинты отвечают 403 (fail-closed). Управление источником, уставки, калибровка
и жизненный цикл эксперимента через REST недостижимы by design.

Для записи в журнал через `POST /api/v1/log` укажите точный `experiment_id`,
если запись относится к конкретному эксперименту. Устаревший ID будет
отклонён. Если ID не указан, запись честно сохраняется как
`experiment_unbound=true`; сервер не подставляет «текущий» эксперимент задним
числом. `request_id` (32 строчных шестнадцатеричных символа), author и source
назначает сервер, а не операторский клиент. Это сохраняет единого владельца
журнала и безопасную идемпотентность повторной доставки.

В live-reading JSON нечисловое или бесконечное значение прибора передаётся как
`null`, но status и identity остаются видимыми. `null` означает «текущее
числовое значение недоступно», а не ноль и не нормальное состояние; оператор
должен опираться на рядом показанные status/stale/disconnected признаки.

## 3. Главное окно

Интерфейс построен на постоянном «шелле»: вкладок больше нет. Экран состоит
из четырёх постоянных элементов и центральной области, куда открываются
рабочие поверхности (оверлеи).

### Верхняя вахтенная панель (top watch bar)

Всегда видна, пять зон:

- **Engine** — `Engine: работает` / `Engine: нет связи`
- **Эксперимент** — имя активного эксперимента, текущая фаза и прошедшее время;
  клик открывает оверлей «Эксперимент». Рядом — бейдж режима: `Эксперимент`
  (нейтральный, архив пишется), `Отладка` (янтарный, архив не создаётся и
  автоматические отчёты не формируются) или `REPLAY` (воспроизведение
  исторических данных). Клик по бейджу переключает режим через диалог
  подтверждения.
- **Контекст** — давление, `Т 2ст.` (Т12, вторая ступень GM-cooler) и
  `Т N₂` (Т11, азотная плита) — метрологически фиксированные опорные каналы;
  при устаревании данных — пометка «(устар.)»
- **Каналы** — сводка `N/M норма`, при наличии — `K вне нормы` и `ожидают`
- **Тревоги** — `Тревоги: N` (клик открывает оверлей «Тревоги»)

### Инструментальная панель (tool-rail)

Вертикальная колонка иконок слева. Слоты сверху вниз:

- `Дашборд` — домашняя поверхность (сводка)
- `Новый эксперимент`
- `Эксперимент` (Ctrl+E)
- `Источник мощности` (Ctrl+K)
- `Аналитика` (Ctrl+A)
- `Теплопроводность` (Ctrl+C) — открывает оверлей автоизмерения; развёртка
  запускается отдельно кнопкой `Старт`
- `MultiLine`
- `Тревоги` (Ctrl+M)
- `Служебный лог` (Ctrl+L)
- `База знаний`
- `Приборы` (Ctrl+D)
- `Ещё` (⋯) — меню: `Архив` (Ctrl+R), `Калибровка`, `Настройки`,
  `Открыть Web-панель`, `Перезапустить Engine`

Активный слот подсвечивается акцентной рамкой слева. Оверлеи открываются
поверх текущей поверхности. Переходные `Ctrl+1…Ctrl+9` продолжают работать
для операторов, привыкших к позициям слотов.

### Центральная область

Показывает либо дашборд, либо открытый оверлей.

### Нижняя строка состояния (bottom status bar)

Пассивный технический ряд:

- safety state — в нижнем регистре, как в логах (`safe_off`, `ready`,
  `running`, `fault_latched`, `manual_recovery`…)
- аптайм лаунчера (`Лаунчер HH:MM:SS`) — время работы операторского
  интерфейса с момента запуска, не engine и не эксперимента
- свободное место на диске
- текущая скорость поступления данных (`изм/с`)
- состояние подключения к engine
- текущее время

На Windows доступна иконка в системном трее. Это coarse-индикатор launcher, а
не дубликат dashboard или списка тревог.

- В обычном рабочем цикле красный сейчас достижим по известному
  `safety_state=fault|fault_latched`. Resolver также предусматривает красный
  при авторитетном ненулевом числе тревог, но launcher пока не получает
  `alarm_count`, поэтому активная тревога сама по себе цвет трея не меняет.
- Жёлтый означает неполную информацию: нет связи, safety state неизвестен,
  данные несвежие, состояние отчётности неизвестно/неисправно либо отсутствует
  авторитетное точное число тревог. Из-за неподключённого `alarm_count` текущая
  launcher-проводка остаётся жёлтой даже при поступающих данных.
- Зелёный предусмотрен resolver только при подтверждённой связи, допустимом
  safety state, свежих данных, исправной отчётности и точном нуле тревог; в
  текущей launcher-проводке он недостижим.

Форма значка и русская подсказка дублируют цвет. Трей не доказывает
safe/ready/recording или отсутствие тревог. Snapshot ingress главного окна уже
подключён; открытым software gate остаётся именно авторитетная передача
`alarm_count` в launcher/tray. При незавершённом shutdown красный имеет
отдельное значение «завершение не окончено» — см. раздел «Завершение launcher».

## 4. Основные поверхности

### 4.1. Дашборд

`Дашборд` (домашний слот ToolRail) — основная операторская сводка:

- safety state
- число активных тревог, ещё не квитированных оператором; acknowledged active
  evidence остаётся в alarm surface до authoritative clear
- короткий Keithley status по каналам A/B
- temperature cards
- pressure strip
- тренды по температуре

Используйте её как первую точку проверки после запуска.

### 4.2. Keithley 2604B

Overlay «Источник мощности» показывает оба канала «Канал А» (`smua`) и «Канал B» (`smub`) бок о бок. Старые ожидания про disable / hide / remove `smub` считать устаревшими.

Per каждый канал:

- поля P цели (Вт), V предела (В), I предела (А) — `QDoubleSpinBox`
- кнопки `Старт`, `Стоп`, `АВАР. ОТКЛ.`
- 4 живых показания: Напряжение (В), Ток (А), Сопротивление (Ом), Мощность (Вт)
- 4 графика V / I / R / P, рулон 10 минут / 1 час / 6 часов (переключатель сверху, общий на оба канала)
- бейдж состояния: `ВЫКЛ`, `ВКЛ`, `АВАРИЯ` — значение приходит из backend-канала `analytics/keithley_channel_state/{smua,smub}`

Panel-level:

- `Старт A+B` / `Стоп A+B` — отправляют соответствующие команды на оба канала
- `АВАР. ОТКЛ. A+B` — один диалог подтверждения, затем аварийное отключение обоих каналов

Поведение спинбоксов:

- спин P / V / I в состоянии `ВКЛ` — через 300 мс «оседания» отправляется `keithley_set_target` (для P) или `keithley_set_limits` (для V/I); рапидные клики сворачиваются в одно сообщение
- в состояниях `ВЫКЛ` / `АВАРИЯ` спины не отправляют команды — канал не принимает питание

Safety gating: при `set_safety_ready(False, reason)` недоступны все управляющие элементы кроме `АВАР. ОТКЛ.` — кнопка аварийного отключения остаётся доступной пока есть связь. Отображается подпись «Управление заблокировано: {reason}».

Состояние канала в GUI всегда следует backend status, не только телеметрии.

Fail-closed safety discipline:

- если OFF не подтверждён readback'ом, система не показывает ложный
  `safe_off`: состояние переходит в `fault_latched`
- после connect() неподтверждённый force-OFF блокирует RUN до следующего
  верифицированного OFF; это блок запуска, а не потеря ручного аварийного
  отключения
- `АВАР. ОТКЛ.` должен восприниматься как команда, требующая подтверждённого
  результата: при неуспешной проверке остаётся FAULT, а не «условно выключено»

TSP watchdog для Keithley выбирается в `config/instruments.yaml`:

```yaml
keithley:
  watchdog:
    mode: "off"          # off | best_effort | required
```

`off` — TSP-скрипт не загружается, safety держит host-side контур.
`best_effort` активирует только late-pet проверку и пишет CRITICAL, что она не
покрывает полную смерть хоста. `required` — fail-closed режим: он требует
отдельный `cryodaq_wdog_autonomous == 1`. Текущий v3 честно сообщает 0,
поэтому прибор считается недоступным и RUN не разрешается. Переход на
`best_effort` — осознанное принятие этой деградации оператором.

Если trip уже защёлкнут, connect сохраняет его и остаётся доступным только для
диагностики/OFF/recovery; RUN заблокирован немедленно. Backend recovery-contract
требует reason и перед очисткой latch повторно подтверждает OFF обоих каналов;
только затем он может очистить улику и реактивировать late-pet check. Но в
текущем GUI reasoned `safety_acknowledge` ещё не выведен: duty operator не должен
посылать raw ZMQ-команду. Выполните emergency OFF/readback и передайте recovery
ответственному инженеру; неуспех или неподтверждённый OFF оставляет FAULT. Для
`required` v3 сначала нужно осознанно выбрать `best_effort`; `off` означает
намеренно отключить весь TSP-path и не считается consume latch.

### 4.3. Тревоги

Оверлей `Тревоги` (слот ToolRail «Тревоги», Ctrl+M) показывает одну
каноническую таблицу Alarm Engine v2. Отдельного v1-ingress и второй таблицы
нет. Активные и подтверждённые срабатывания сохраняют точную идентичность
экземпляра engine и activation; устаревшее ACK не может подтвердить новое
срабатывание с тем же именем.

**Уровни внимания** отображаются текстом, формой и цветом: fault/critical —
`КРИТ`, единая caution-ступень — `ВНИМ`. Legacy `warning` принимается только
как совместимый вход и отображается той же caution-ступенью, а не отдельным
уровнем. Неизвестная тяжесть трактуется консервативно как fault.

**Квитирование** доступно, когда GUI считает соединение live, панель не
находится в replay/read-only режиме, последний принятый Alarm v2 snapshot
остаётся авторитетным и у строки есть точные `engine_instance_id` и
`activation_id`. Отдельного age/TTL-gate для alarm snapshot сейчас нет: само
истечение времени не отключает ACK, а snapshot с меньшей revision игнорируется
без отзыва ранее принятой authority. Поэтому это нельзя описывать как проверку
«свежести» конкретной тревоги. Backend повторно проверяет точную идентичность и
отклоняет устаревшее или заменённое срабатывание.

GUI отправляет пустые `operator` и `reason`. ACK переводит точное срабатывание
в состояние `acknowledged`, но не является доказательством того, какой именно
оператор принял ответственность и по какой причине. Он не снимает hazard, не
очищает condition и не меняет safety authority.

Disconnect, replay/read-only, malformed/unidentified snapshot или отсутствие
точной идентичности блокируют действие, сохраняя последние строки и
provenance. Пока авторитетный backend не сообщил `cleared`, acknowledged active
evidence остаётся в операторском пути.

Backend transitions: `activated`, `acknowledged`, `cleared`. Пока авторитетный
backend не сообщил `cleared`, условие остаётся unresolved; acknowledged active
evidence не исчезает из операторского пути.

### 4.4. Служебный лог (Журнал оператора)

Overlay «Служебный лог» открывается через ToolRail слот «Служебный лог» (Ctrl+L). Основной сценарий — приёмка / сдача смены.

**Composer (верхняя карточка):**

- поле «Автор» — имя оператора, запоминается между запусками GUI (`QSettings` ключ `last_log_author`)
- поле «Теги» — через запятую, пример: `shift, handover, alarm`
- поле сообщения — произвольный текст записи
- чекбокс «Привязать к текущему эксперименту» — автоматически отмечается когда есть активный эксперимент; отключён когда эксперимента нет
- кнопка «Сохранить» — отправляет запись в engine; при успехе поле сообщения очищается, автор сохраняется, запись добавляется в таймлайн

**Фильтры (средняя карточка):**

- chips (взаимоисключающие): `Все` / `Текущий экспт.` / `Последние 8ч` / `За сутки`. По умолчанию активен `Последние 8ч` — сценарий «что было за мою смену»
- поле «Поиск» — подстрока в тексте записи, case-insensitive, debounce 250 мс
- поле «Автор» — точное совпадение автора, case-insensitive
- поле «Тег» — одиночный тег, проверка на членство в списке тегов записи

Комбинация фильтров работает по AND: например, `Автор=Владимир` + `Тег=alarm` + `Поиск=клапан` — все три условия применяются одновременно.

**Timeline (нижняя карточка):**

- записи группируются по календарному дню (локальное время) с заголовком-разделителем
- каждая запись: время (HH:MM, моноширинный шрифт с tabular figures), автор, текст, chips с `experiment: <id>` и `tag: <name>`
- системные записи (`author == "system"`) — приглушённый серый, отличаются визуально от операторских

**Footer:**

- счётчик «Загружено: N записей · отфильтровано: M»
- кнопка «Загрузить ещё 50» — увеличивает серверный лимит на 50 и перезагружает

**При потере связи с engine:**

- composer отключается, появляется баннер «Нет связи с engine»
- таймлайн остаётся читаемым (данные устарели, но контекст важнее)
- при восстановлении связи composer автоматически возвращается

**QuickLogBlock на дашборде** — это отдельная, упрощённая поверхность для быстрых однострочных заметок без фильтров. Журнал оператора и QuickLogBlock дополняют друг друга, не заменяют.

Записи сохраняются в SQLite и могут попасть в отчёт.

### 4.5. Архив

Overlay «Архив экспериментов» открывается через ToolRail: «Ещё» → «Архив» (Ctrl+R).

**Фильтры (верхняя карточка):**

- выпадающий список «Шаблон» — по умолчанию «Все»
- текстовые поля «Оператор» и «Образец» — точное совпадение на сервере
- поля «С» / «По» — диапазон дат, по умолчанию 30 дней от сегодня
- фильтр «Отчёт»: `Все` / `Есть отчёт` / `Нет отчёта`
- сортировка: `Сначала новые` / `Сначала старые` / `Оператор А-Я` / `Образец А-Я`
- кнопка `Обновить` — перезагрузить список

**Таблица (2/3 ширины):**

- 9 колонок: Начало, Конец, Эксперимент, Шаблон, Оператор, Образец, Статус, Отчёт, Данные
- колонки времени в моноширинном шрифте с tabular figures для выравнивания по разрядам
- колонки «Отчёт» и «Данные»: `Да` если соответствующий артефакт присутствует, `—` если нет
- при клике по строке — панель сведений справа обновляется атомарно
- по умолчанию после загрузки выделяется первая (самая новая) запись

**Сведения (1/3 ширины):**

- заголовок эксперимента, ID, статус
- метаданные: шаблон, оператор, образец, диапазон дат, папка артефактов, пути PDF / DOCX
- заметки (только чтение)
- состав: `runs=N, artifacts=M, результаты=K`
- списки прогонов / артефактов / результатов
- кнопки действий: `Папка` (открыть директорию), `PDF` (открыть raw PDF), `DOCX` (открыть редактируемый DOCX), `Перегенерировать` (запустить пересборку отчёта)

Роль артефакта отображается ASCII-тегом: `[ДАННЫЕ]` (experiment_data), `[ИЗМЕРЕНИЯ]` (measured_values), `[УСТАВКИ]` (setpoint_values).

Cold storage rotation прозрачен для оператора: старые daily SQLite
переносятся в zstd Parquet под `data/archive`, но история, архив,
экспорт CSV / HDF5 / Excel, отчёты и служебный лог читают hot ∪ cold
через общий архивный слой. Если день уже ротирован, отсутствие
`data/data_YYYY-MM-DD.db` само по себе не означает потерю данных.

**Экспорт данных (нижняя карточка, K6):**

Кнопки `CSV...` / `HDF5...` / `Excel...` запускают полный экспорт из SQLite (весь временной ряд, не только выбранный эксперимент):

- **CSV** — один файл, выбор через `QFileDialog`
- **HDF5** — папка с файлами `data_YYYY-MM-DD.h5`, выбор через `QFileDialog.getExistingDirectory`
- **Excel** — один файл `.xlsx`, выбор через `QFileDialog`

Экспорт выполняется в фоновом потоке (`QThread`) — GUI не блокируется, даже если экспорт занимает минуты. Пока один экспорт идёт, все три кнопки отключены, чтобы избежать конкуренции за SQLite-файлы. Статус-баннер сообщает о прогрессе и результате.

**При потере связи с engine:**

- `Обновить` / `Перегенерировать` / экспорт — отключены
- открытие папки / PDF / DOCX — остаётся доступным (локальная файловая система)
- баннер показывает статус соединения

Отсутствующий файл отчёта показывается явно. GUI не должен делать вид, что отчёт есть, если файла нет.

### 4.6. Калибровка

Overlay «Калибровка датчиков» открывается через ToolRail: «Ещё» → «Калибровка». Трёхрежимный workflow с автопереключением по статусу engine:

**Setup (по умолчанию):**

- `Опорный канал` — выпадающий список каналов LakeShore (формат `<instrument>:<channel>`, например `LS218_1:Т1 Криостат верх`). Обычно выбирается метрологически фиксированный Т11 или Т12.
- Группы целевых каналов (по одной на LakeShore прибор), по умолчанию все отмечены. Опорный канал автоматически исключается на этапе старта — не нужно снимать галочку вручную.
- `Начать калибровочный прогон` создаёт эксперимент с `template_id=calibration` и запускает сбор данных.
- Карточка `Импорт внешней кривой` — две кнопки (`.340` / `JSON`). Клик → файловый диалог → engine вызывает `calibration_curve_import` → баннер с `curve_id` при успехе, таблица существующих кривых обновляется. Формат `.330` более не поддерживается (удалён 2026-04-25).
- Таблица `Существующие кривые` — 5 колонок (Датчик / Curve ID / Зон / RMSE / Источник), обновляется при подключении и после успешного импорта.

**Acquisition (автоматически после старта):**

- Карточка статистики: имя эксперимента, прошедшее время (HH:MM:SS), количество записанных точек, диапазон опорной температуры.
- `Покрытие по температуре` — горизонтальный бар плотности записей: dense (зелёный), medium (жёлтый), sparse (оранжевый), empty (серый). Операторская задача — добиться dense от start до end температуры.
- Карточка `Последние значения` — последние 5 `_raw` / `sensor_unit` показаний в моноширинном шрифте.
- Запись идёт автоматически. Дождитесь полного cooldown, затем завершите эксперимент стандартным путём («Эксперимент» → «Остановить запись»).

**Results (автоматически после завершения):**

- Выпадающий список каналов из только что завершённой калибровки.
- `Метрики подгонки`: количество raw пар, downsampled точек, breakpoints, зон Chebyshev, RMSE, Max ошибка — обновляются при выборе канала.
- Карточка `Экспорт кривой` — 4 кнопки (`.cof` / `.340` / `JSON` / `CSV`). Клик → файловый диалог → engine пишет ВСЕ четыре формата за один вызов (выбранный формат кладётся по указанному пути, остальные три — в engine-default локации). `.cof` — портативный текстовый формат с сырыми коэффициентами Chebyshev по зонам, пригодный для re-evaluation через `numpy.polynomial.chebyshev` без зависимости от CryoDAQ.
- Карточка `Применить в CryoDAQ`:
  - Чекбокс `SRDG + calibration curves (глобально)` — при отметке сначала вызывается `calibration_runtime_set_global` с `on`.
  - Выпадающий список `Политика канала`: `Наследовать` (от глобального), `Включить`, `Выключить`.
  - Кнопка `Применить` запускает цепочку `calibration_curve_lookup` → `calibration_runtime_set_channel_policy` с `channel_key`, `policy`, `sensor_id` + резолвленным `curve_id`.
  - Канал должен быть выбран — иначе баннер с ошибкой.

**При отсутствии связи с engine:**

- кнопки Setup / Import / Export / Apply отключены
- 3-секундный опрос режима останавливается (auto-switch не произойдёт)
- при восстановлении связи — всё возобновляется, таблица кривых автоматически обновляется

Runtime calibration поддерживает глобальный режим `on/off` и per-channel policy. При отсутствии curve, assignment, `SRDG` или ошибке вычисления backend консервативно возвращается к `KRDG` и логирует причину. Поведение на живом LakeShore требует отдельной lab verification.

### 4.7. Приборы

Overlay `Приборы` — пре-эксперимент проверочный экран. Две секции:
карточки приборов (сверху) и диагностика датчиков (снизу). Оператор
открывает его перед любым измерением, чтобы убедиться: каждый прибор
на связи, каждый датчик здоров.

**Карточки приборов.** Одна карточка на instrument_id. В шапке —
круглый индикатор (`STATUS_OK` / `STATUS_CAUTION` / `STATUS_FAULT` /
`STATUS_STALE`) и имя прибора. Ниже: статус («Норма / Нет данных /
Нет связи / Предупреждение»), время последнего ответа, счётчики
показаний и ошибок. Таймаут живости адаптивный: `timeout = медиана
интервалов × 5` (минимум 10 с, по умолчанию 300 с). Если конкретный
прибор теряет связь, это отражается на его карточке независимо от
остальных.

**Диагностика датчиков.** 7-колоночная таблица (Канал / T(K) / Шум /
Дрейф / Выбросы / Корр. / Здоровье) с суммарным бейджем в правом
верхнем углу (чипы «N ОК / N ПРЕД / N КРИТ»). Engine опрашивается
командой `get_sensor_diagnostics` каждые 10 с; строки сортируются по
health_score по возрастанию (худшие сверху). Пороги health: ≥80 — ОК
(зелёный), 50–79 — предупреждение (янтарный), <50 — критично
(красный). Row background окрашен соответствующей полупрозрачной
заливкой DS токена.

При отключении engine опрос диагностики ставится на паузу, но последние строки
и карточки остаются видимыми с возрастающей liveness/stale индикацией. Это
evidence-retaining, action-fail-closed поведение: сохранённые данные не дают
новой authority и не превращаются в current truth.

### 4.8. Теплопроводность

Overlay «Теплопроводность» открывается через ToolRail слот «Теплопроводность». Основной сценарий — измерение теплопроводности методом стационарной разности температур при ступенчатом повышении мощности.

**Выбор цепочки датчиков (левая карточка):**

- чекбоксы со всеми видимыми T-каналами из ChannelManager
- порядок выбора = порядок в цепочке: первый чекбокс = горячий конец (рядом с источником мощности), последний = холодный
- кнопки `↑` / `↓` перемещают сфокусированный датчик по цепочке
- выпадающий список «Источник P»: `Keithley_1/smua/power` или `Keithley_1/smub/power`
- кнопка `Экспорт CSV` выгружает текущий снимок табличных значений (с прогнозом) в файл

**Live-показания (правая карточка):**

- Steady-state баннер над таблицей: `ГОТОВО — стационар достигнут` при
  min_settled ≥ 99%, иначе `Стабилизация N% — ещё ~M мин`. Порог баннера 99%
  является только presentation-индикатором; переход шага авторазвёртки
  использует отдельно настроенные `Порог стабилизации` и `Мин. ожидание`.
- ряд индикаторов: стабильность (`Нестабильно` при `|dT/dt| > 0.01 К/мин`,
  иначе `Стабильно`) и текущая мощность P
- таблица R/G: 11 колонок (Пара, T гор./хол., dT, R, G, T∞ прогноз, τ, Готово %, R/G прогноз) + строка `ИТОГО` с суммарными значениями по всей цепочке
- график температур со времени начала + прогнозная зона (1/3 истории, padding 2%)

**Автоизмерение (нижняя карточка):**

Параметры:
- `Начальная P` (Вт) — стартовая мощность развёртки
- `Шаг P` (Вт) — приращение между шагами
- `Шагов` — количество точек в развёртке
- `Порог стабилизации` (%) — percent_settled предсказателя, при котором шаг считается завершённым (по умолчанию 95%)
- `Мин. ожидание` (с) — минимальное время до проверки стабилизации (защита от ложного срабатывания на начальном переходном процессе, по умолчанию 30 с)

Под параметрами — preview списка мощностей, рассчитанного из Start / Step / Count.

Алгоритм развёртки:

1. `Старт` проверяет live-связь, наличие ≥ 2 датчиков и параметры, формирует
   power list, переводит FSM в `stabilizing` и отправляет первый
   `keithley_set_target`.
2. Пока ответ на текущую команду не получен, timer не продвигает шаг. Ошибка,
   timeout или потеря связи сохраняют `stabilizing`, останавливают timer и
   показывают, что исход автоизмерения неизвестен.
3. После подтверждённой установки цели каждый tick проверяет `Мин. ожидание` и
   минимум `percent_settled`. При выполнении обоих условий точка записывается и
   отправляется следующая цель.
4. После последней точки GUI отправляет `keithley_stop`. FSM переходит в `done`
   и показывает сводку только после успешного ответа текущего поколения,
   подтверждающего отключение источника.
5. `Стоп` также только отправляет `keithley_stop` и ожидает подтверждения.
   Состояние `idle` и баннер «отключение подтверждено» появляются лишь после
   успешного ответа; нажатие кнопки само по себе не является доказательством
   OFF.

**Перед финализацией эксперимента:** текущий ExperimentOverlay не вызывает
`is_auto_sweep_active()`/`get_auto_state()` и автоматически не блокирует
финализацию при активном или outcome-unknown автоизмерении. Оператор должен
вручную нажать `Стоп` и дождаться явного «отключение подтверждено» либо
успешного завершения со state `done`. Доступная кнопка `Завершить эксперимент`
не доказывает, что Keithley выключен; автоматический finalize guard остаётся
открытым software/safety gate.

**Flight recorder:**

Каждый настроенный 1 Hz tick overlay пишет 18-колоночный CSV в
`<data_dir>/conductivity_logs/conductivity_<timestamp>.csv` (encoding
`utf-8-sig` для открытия в Excel с кириллицей). Запись идёт автоматически при
цепочке ≥ 2, независимо от состояния автоизмерения, и файл закрывается в
`closeEvent`. Это дополнительный GUI flight-recorder для post-hoc анализа, а
не авторитетная lossless acquisition-запись: GUI timer может задержаться или
пропустить tick, а первичными остаются persisted engine/archive readings.

**При потере связи с engine:**

- при активной развёртке отключаются и `Старт`, и `Стоп`: без live-связи GUI
  не имеет authority утверждать, что команда отключения доставлена;
- состояние остаётся `stabilizing`, timer останавливается, а исход помечается
  неизвестным;
- последние таблица и график сохраняются; выбор цепочки и экспорт CSV остаются
  доступны как локальные действия;
- после восстановления связи `Стоп` становится доступным для нового
  авторитетного запроса OFF, но `Старт` остаётся заблокированным до успешного
  подтверждения `keithley_stop`. Только после него состояние возвращается в
  `idle`.

## 5. Эксперименты

Главная operator workflow должна различать режимы `Эксперимент` и `Отладка`.

В режиме `Отладка` не должны:

- создаваться архивные карточки экспериментов
- запускаться автоматические внешние отчёты по эксперименту

Текущий режим виден в верхней панели в виде badge'а. Жёлтая ОТЛАДКА требует
явного внимания — данные собираются, но архивная запись не создаётся.

В верхней части дашборда находится компактный phase strip:
номер и название текущей фазы, длительность, ETA и ключевые
метрики inline (R_thermal, давление). Подробная аналитика —
в overlay Аналитика.

Overlay `Эксперимент` работает в режиме connection-gated: при отсутствии
свежих данных от engine кнопки действий — «Сохранить карточку»,
«Завершить эксперимент», «← предыдущая фаза», «следующая фаза →» —
автоматически блокируются. Сами данные (фазы, длительности, timeline)
при этом остаются видимыми: оператор сохраняет картину состояния даже
при разрыве связи. Восстановление соединения возвращает кнопки в
работоспособное состояние.

Переключить режим можно кликом по бейджу в верхней вахтенной панели.
Появится диалог подтверждения с описанием последствий.

Новый эксперимент создаётся кнопкой `Новый эксперимент` в ToolRail или из
дашборда; текущий эксперимент открывается через оверлей `Эксперимент`.
Завершение выполняется кнопкой `Завершить эксперимент` в этом оверлее.

Connection gate этой кнопки не проверяет состояние автоизмерения
«Теплопроводность»; перед финализацией выполните процедуру подтверждённого OFF
из §4.8.

Система использует experiment templates из `config/experiment_templates/*.yaml`.

При старте эксперимента оператор выбирает шаблон и вводит метаданные. После завершения сохраняются:

- `metadata.json`
- операторские записи
- артефакты отчёта

## 6. Отчёты

Целевой внешний отчётный контракт:

- `report_raw.pdf` — автоматически сгенерированный сырой файл
- `report_editable.docx` — редактируемая версия для оператора

Основой генерации отчёта служат архивная карточка эксперимента и её артефакты; для части данных текущая реализация всё ещё может использовать fallback-чтение из SQLite.

Гарантированный артефакт:

- `report_editable.docx`

Опциональный артефакт:

- `report_raw.pdf`

PDF-конвертация остаётся best-effort и зависит от внешнего `LibreOffice` / `soffice`.

## 7. Типовой рабочий сценарий

### Начало эксперимента

1. Запустите `engine`
2. Откройте GUI
3. Проверьте `Дашборд` и `Тревоги`
4. Убедитесь, что нет необработанных critical conditions
5. Нажмите `Новый эксперимент` в ToolRail и заполните карточку
6. При необходимости внесите заметку в `Служебный лог`

### Работа с Keithley

1. Откройте `Источник мощности` (ToolRail, Ctrl+K)
2. Выберите нужный канал A или B
3. Запустите канал
4. Контролируйте backend-driven status и показания
5. При необходимости используйте channel-scoped stop или emergency off

### Завершение эксперимента

1. Остановите эксперимент в оверлее `Эксперимент`
2. Проверьте, что архивная запись появилась в оверлее `Архив` (ToolRail: «Ещё» → «Архив», Ctrl+R)
3. При необходимости сформируйте или пересоберите отчёт

## 8. Что считать нормальным, а что нет

Нормально:

- GUI закрыт, а engine продолжает писать данные
- в архиве нет PDF, но есть DOCX
- GUI может быть закрыт, пока `engine` продолжает собирать данные
- В архиве может быть DOCX без PDF
- Runtime calibration может консервативно вернуться к `KRDG`, если недоступны curve, assignment или вход `SRDG`
- Tray может показывать warning, если у GUI недостаточно backend truth
- старые daily DB уже перенесены в `data/archive`, а история и архив всё равно показывают эти дни

Ненормально:

- GUI показывает healthy / ON без backend status
- Архив утверждает, что отчёт существует, хотя файла нет
- Export запускает backend-работу после отмены file dialog
- Calibration panel скрывает runtime fallback или отказ и показывает оптимистичный applied state
- RUN разрешается после неподтверждённого OFF или при недоступном required-watchdog

## 9. Тема интерфейса

Меню **Настройки → Тема** динамически перечисляет валидированные пакеты из
`config/themes/*.yaml`; `warm_stone` остаётся значением по умолчанию. Поэтому
руководство намеренно не фиксирует изменчивое количество тем. Галочка всегда
остаётся на теме, реально загруженной в текущем процессе. Выбор другого пакета
валидируется и атомарно сохраняется для следующего обычного запуска; отдельная
неактивная строка меню показывает отложенный вариант.

Выбор темы не останавливает GUI ingress, assistant, bridge, engine, запись или
сбор данных и не освобождает launcher lock. Новая тема появится только после
следующего обычного запуска приложения. Если настройку сохранить не удалось,
текущая тема и весь runtime продолжают работать, а оператор получает явное
сообщение об ошибке без приватных деталей пути.

Операторская safety-лестница одинакова во всех темах: safe → caution → fault,
плюс отдельные stale/disconnected/unavailable состояния. Compatibility-вход
`STATUS_WARNING` является точным alias для `STATUS_CAUTION` и не создаёт
четвёртую ступень. Меняются только нейтральные поверхности и accent; safety
семантика и non-color cues не меняются со стилем.

### Восстановление при неработоспособном GUI

Если после смены темы интерфейс не поднимается или выглядит нечитаемо,
откройте `config/settings.local.yaml` в текстовом редакторе и либо
удалите ключ `theme:`, либо выставьте `theme: warm_stone`. Следующий
запуск прочитает значение по умолчанию. Тот же файл можно удалить
полностью — он будет пересоздан.

Если YAML повреждён или содержит невалидные значения, лоадер
автоматически откатывается к `warm_stone` и пишет предупреждение в лог
(запись в `logs/`). Ни одна ошибка разбора темы не блокирует запуск GUI.

### Завершение launcher

При выборе выхода launcher сначала прекращает новый ingress и автоматические
перезапуски, затем ограниченно по времени завершает принадлежащие ему процессы,
потоки и IPC-очереди. Значок в трее остаётся видимым, пока завершение не
доказано. Если ресурс не завершился, трей становится красным и сообщает
«завершение не окончено»; окно приложения и single-instance lock остаются
активными, а launcher автоматически повторяет только незавершённые шаги.

Исчезновение приложения не должно скрывать продолжающий работать дочерний
процесс. Если красное состояние не снимается после повторных попыток,
зафиксируйте `launcher.log` и состояние процессов для разбора; не запускайте
второй экземпляр поверх удерживаемого lock.

## 10. Известные ограничения

- PDF-конвертация остаётся best-effort и зависит от внешнего `LibreOffice` / `soffice`.
- **Replay engine — обнаружение конфликта портов (loopback only):** детектор занятости портов
  проверяет только `127.0.0.1` и `0.0.0.0`. Если real engine и replay engine привязаны к разным
  сетевым интерфейсам, коллизия не будет обнаружена заранее — вы получите штатные ошибки ZMQ
  EADDRINUSE. Для single-machine лаборатории: всегда останавливайте real engine перед запуском
  replay engine. Флаг `--force-replay` пропускает Python-проверку, но НЕ освобождает порт — он
  нужен только если Python-проверка даёт ложный сигнал при реально свободном порте.

- **Прогноз охлаждения неактивен в mock-режиме:** mock транслирует стационарные температуры
  (уже охлаждённое состояние). `CooldownDetector` активируется только при активном охлаждении
  (dT/dt < −5 K/ч в течение 10 мин). В mock-режиме детектор остаётся в состоянии IDLE,
  буфер не заполняется. С v0.54.0 виджет «Прогноз охлаждения» в этом случае показывает
  горизонтальную линию `T∞` + бейдж «Стационарное состояние ≈ X K» вместо пустой заглушки —
  через `SteadyStatePredictor`, тот же приём, что и у `R тепл.` (ждёт ~10 мин стабильных
  данных до сходимости, порог `percent_settled ≥ 30 %`). Для валидации полноценного прогноза
  охлаждения используйте replay mode (v0.53.0+), который воспроизводит реальные траектории.

## 11. Etalon MultiLine: continuous mode и burst capture

С v0.55.11 драйвер `etalon_multiline` поддерживает два режима работы.

### Когда какой режим

- **`averaged` (по умолчанию)** — request / response: на каждый scheduler
  tick драйвер отправляет `latestlengthvalid` и парсит один cycle. Подходит
  для обычного STE-цикла, где interferometric drift важна на масштабе
  минут / часов и достаточно 1 reading per scheduler poll. Не требует
  предварительной настройки в `MultiLine.exe`.
- **`continuous`** — после connect драйвер шлёт `startmeasnogui` и держит
  background listener: сервер сам пушит `channeldata_` cycle после каждого
  measurement (cycle rate определяется FPGA + analysis step и empirically
  ~1–10 Гц). Драйвер decimates до `target_rate_hz` (по умолчанию 1.0)
  перед публикацией Reading в ZMQ — operator видит ту же 1 Hz скорость
  что и в averaged, но при этом доступен **burst capture**: моментальная
  запись raw cycles на максимальной hardware cadence в Parquet blob.

### Pre-conditions для continuous mode

В `MultiLine.exe`:
1. Каналы выбраны и сгруппированы (`isgrouped=1`).
2. Окно измерения закрыто (driver не открывает GUI на сервере).
3. Setup сохранён в software (lasers ready / aligned).

Если pre-conditions не выполнены, сервер отклонит `startmeasnogui` либо
не пришлёт ни одного cycle. Driver логирует empirical first-cycle latency
на INFO (`MultiLine 'Multiline_1' first cycle received after %.2fs`) —
если в логах нет этой строки в течение ~10 секунд после connect, проверьте
software setup на сервере.

### Как переключить

Режим переключается через `config/instruments.yaml` (или
`instruments.local.yaml` для деплой-специфичного override):

```yaml
- type: etalon_multiline
  name: "MultiLine_1"
  host: "192.168.1.50"
  port: 2001
  channels: [1, 2, 3, 4]
  mode: continuous           # вместо averaged
  target_rate_hz: 1.0        # 0.01 .. 10 (1.0 default)
```

Изменение требует **рестарта engine** — runtime hot-reload не поддерживается.

### Workflow: burst capture

В overlay «MultiLine» снизу под env строкой добавлена строка
«Захват вибрации»:
- spinbox: длительность (1..600 с, по умолчанию 10),
- кнопка «Записать» (превращается в «Остановить» во время записи),
- mono статус справа.

При клике «Записать»:
1. Engine вызывает `driver.burst_start(experiment_id=<active>)`.
2. Listener продолжает receive cycles в обычном темпе, но теперь
   КАЖДЫЙ cycle ALSO добавляется в burst buffer (decimation для emit
   в ZMQ остаётся, full cadence только в blob).
3. По истечении duration или ручной нажатии «Остановить», engine
   вызывает `driver.burst_stop()` — flush buffer в Parquet.
4. Путь к файлу появляется в статусе:
   - Если active experiment: `data/experiments/<id>/multiline_burst_<utc_iso>.parquet`.
   - Без эксперимента: `data/multiline_bursts/<utc_iso>.parquet`.

Parquet схема включает все 17 полей `channeldata` per row (length,
env T/P/RH, intensity bounds, все 10 error flags). Этого достаточно
для post-hoc PSD/Allan deviation анализа в Jupyter.

Если burst capture запрошен в `mode: averaged`, engine отвечает
ошибкой «Burst capture requires continuous mode» — статус строка
отображает её дословно.

### Гарантии persistence

- Decimated Reading из continuous mode проходит **тот же**
  `Scheduler.write_immediate → SQLiteWriter → DataBroker` путь что и
  averaged — SQLite write завершается **до** broker publish, никаких
  изменений в persistence-first инвариант не вносится.
- Burst Parquet — отдельный артефакт, пишется через `asyncio.to_thread`
  чтобы не блокировать scheduler tick. Если operator закроет engine
  во время активного burst, buffer **не** persist'ится — explicit
  `burst_stop` is the only save path.

## 12. База знаний

Оверлей `База знаний` открывается из основного ToolRail между `Служебный лог` и
`Приборы`. Поиск, встроенный помощник и индекс принадлежат отдельному процессу
`cryodaq-assistant`, а не engine. Launcher запускает этот процесс, когда
соответствующая функция требуется конфигурацией; отдельный запуск
`cryodaq-engine` + `cryodaq-gui` помощника не предоставляет.

Результаты поиска и ответы помощника — advisory observational text. Они могут
быть неполными или устаревшими и не подтверждают safety state, alarm clearance,
source OFF, readiness или acceptance. Проверяйте их по авторитетному GUI/backend
и утверждённым процедурам.

### Что в базе

В состав корпуса при bootstrap или явно разрешённом rebuild входят:

- `data/knowledge/equipment_manuals/` — установленная PDF-документация
  оборудования (например, Etalon MultiLine, LakeShore, Keithley и Thyracont);
  точный состав и объём зависят от локального deployment.
- `data/knowledge/procedures/` — markdown процедуры (захолаживание,
  аварийное отключение, troubleshooting). Subdir → category.
- `data/knowledge/reference/` — operator_manual, README, README.en,
  CHANGELOG (последний — section-aware по версиям).
- `data/experiments/<id>/metadata.json` — past experiments.
- `data/data_*.db` operator_log table — журнал оператора.

### Добавление источников и перестроение индекса

Добавленный PDF или Markdown не появляется автоматически в уже заполненном
индексе. Пустой индекс bootstrap-ится при старте `cryodaq-assistant`;
заполненный индекс при повторном старте не перестраивается.

В текущем checkpoint кнопка `Обновить индекс` существует и отправляет
мутационную команду `rag.rebuild_index` процессу assistant. Это противоречит
действующей observational-only границе помощника, поэтому кнопка пока не
является одобренной live-операцией и не должна использоваться во время
эксперимента. Требуется отдельно утверждённый административный/offline путь
перестроения индекса.

Restart engine не запускает и не перестраивает assistant index и может
прервать acquisition. Restart launcher может заново поднять настроенный
assistant, но не гарантирует rebuild заполненного индекса и также не является
процедурой индексации.

До закрытия authority gate передайте новый PDF или процедуру ответственному
инженеру для offline/admin rebuild и проверьте результат до следующего
эксперимента. PDF размещается в
`data/knowledge/equipment_manuals/<имя>.pdf`; Markdown-процедура — в
`data/knowledge/procedures/<имя>.md`, где первая строка `# Заголовок процедуры`
задаёт title, а подкаталоги задают category.

### Цитирование источников

`prettify_source_label` производит operator-readable citation labels:

| source_kind         | пример                                |
| ------------------- | ------------------------------------- |
| equipment_manual    | `Etalon MultiLine — стр. 5`           |
| procedure           | `Процедура: Аварийное отключение`     |
| operator_manual     | `Operator Manual`                     |
| readme / readme_en  | `Project README` / `Project README (EN)` |
| changelog           | `CHANGELOG v0.55.7`                   |
| experiment_metadata | `Эксперимент: Cooldown S-001 — 2026-04-15` |
| operator_log        | `Журнал: Vladimir — 2026-04-15`       |

Такие же labels показываются в карточках результатов, чтобы оператор
быстро отличал руководство прибора, процедуру, changelog и журнал.

Документ описывает текущее реализованное состояние и не обещает будущие функции.
