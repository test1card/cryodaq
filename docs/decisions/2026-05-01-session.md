## 14:30 — v0.47.3 display name resolution with LATE BINDING

### Thesis
Operators rename channels mid-campaign via GUI ChannelEditor.
Engine restart destroys state. IntentClassifier was caching
ChannelManager hint at construction → renames invisible until
restart.

### Reasoning
LATE BINDING pattern: any component using ChannelManager-derived
data MUST read fresh state per call, not cache at construction.
Working precedent: `ChannelStateTracker` (alarm_v2 path) already
rebuilds `_short_to_full` from live readings.

### Decision
- `IntentClassifier.classify()` rebuilds prompt with current
  ChannelManager hints per call.
- `ChannelManager.find_by_name()` exact + substring + Latin/Cyrillic
  fallback.
- `QueryRouter._resolve_target_channels()` uses fresh ChannelManager
  per query.

### Consulted
Codex audit PASS.

## 17:00 — v0.47.4 channel ID normalization (Option B)

### Thesis
`instruments.yaml` driver labels embed display names ("Т7
Детектор"). ChannelManager keys use short form ("Т7"). BrokerSnapshot
storing long key meant `snapshot.latest("Т7")` returned None →
bot answered "Пока не вижу" for all temperature queries.

### Reasoning
Option A (rewrite `instruments.yaml` to emit short keys + migrate
141k SQLite rows + update interlocks regex) — high risk, breaks
safety-critical interlocks regex pattern. Option B (BrokerSnapshot
normalizes at ingest, splits "Т7 Детектор" → "Т7") — pattern already
proven in `ChannelStateTracker._short_to_full` (alarm_v2 path).
Locality of change is the win: one boundary handles two configs
that disagree.

ORCHESTRATION §v1.5.5 rule extracted from this: when configs
overlap, look for normalizing boundary first before structural
rewrite.

### Decision
Option B. `BrokerSnapshot._normalize_channel_id()` at ingest.
19 new tests in `test_channel_id_normalization.py`. Bonus:
ChannelEditor renames now propagate end-to-end through driver →
broker → display chain (LATE BINDING unblocked).

## 19:30 — F27 composition photos ship (v0.50.0)

### Thesis
Operator photographs experimental composition (sample mounting,
sensor placement, cryostat layout) on phone. Currently photos
live on phone, not attached to scientific record. Composition
photos critical for reproducibility ("how was sample mounted in
experiment-042?").

### Reasoning
- Telegram bot photo handler: existing handler ignores photos.
  Natural extension point — `_fetch_updates` already iterates
  message types.
- Photo storage: `<artifact_dir>/composition/` over global
  `~/data/photos/<exp_id>/` — locality (cleans up with experiment
  archival), no separate retention policy needed.
- Confirmation flow: inline keyboard ([Да] / [Нет] / [Другой])
  over auto-attach to active experiment — operator may have just
  finalized, may want different experiment, may have sent by
  mistake.
- ZMQ event `experiment.photo_attached` over GUI polling — same
  pattern as existing `experiment.finalized` /
  `experiment.phase_changed` events. GUI auto-refreshes.
- Caption parsing channel mention: LATE BINDING (fresh
  ChannelManager per call) per §v1.5.2 invariant. Sidecar JSON
  records resolved channel IDs at attach time, but classifier
  re-resolves if needed later (renames respected).
- Pillow dependency: needed for image validation (dimensions,
  corruption check). Approved by architect — adds one
  well-maintained dep in service of corruption rejection.
- Version bump 0.47.4 → 0.50.0: skip 0.48.0 / 0.49.0 reserved for
  F-X / F-Y. F27 was independent track, ready first.

### Decision
Ship as v0.50.0. Tag at branch tip (217b9d40), merge to master via
ort strategy (123e84df). 50 new tests (preliminary +47 to
collected baseline).

minimum). One MEDIUM finding fixed pre-tag (caption sanitization
edge case).
