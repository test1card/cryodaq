# Диагностика неисправностей CryoDAQ

Дерево «симптом → проверка → исправление» для самых частых отказов.
Каждый пункт привязан к реальному поведению кода — не догадка. Общий
источник логов: `logs/` (engine + launcher); там же ищите точные строки
из этого документа.

Если проблема не описана здесь — см. `docs/operator_manual.md` (раздел
«Что считать нормальным, а что нет») и `docs/lab_verification_checklist.md`
для проверок, требующих физического доступа к железу.

---

## 1. Engine не запускается (падает сразу после старта)

<!-- src/cryodaq/engine.py: ENGINE_CONFIG_ERROR_EXIT_CODE = 2, sys.exit paths -->

**Симптом.** `cryodaq-engine` завершается почти мгновенно; лаунчер не
делает автоматических попыток перезапуска.

**Проверка.** Код возврата процесса — `2`. В логе строка вида:

```
CONFIG ERROR (YAML parse): ...
CONFIG ERROR (file not found): ...
CONFIG ERROR (safety|alarm|interlock|housekeeping|channel config): ...
```

**Причина.** Ошибка чтения/валидации конфигурации на старте
(`config/*.yaml`) неустранима повтором — движок сознательно выходит с
отдельным exit-кодом, чтобы лаунчер не крутил бесконечный restart-loop
поверх сломанного конфига.

**Исправление.** Открыть указанный в логе файл (`config/safety.yaml`,
`config/alarms_v3.yaml`, `config/interlocks.yaml`,
`config/housekeeping.yaml`, `config/channels.yaml` — по метке в
сообщении) и исправить синтаксис/значения. После правки — кнопка
«Перезапустить Engine» в лаунчере (автоперезапуск на этом коде
намеренно отключён).

## 2. Лаунчер показывает баннер «Engine недоступен» + звуковой аларм

<!-- src/cryodaq/launcher.py: _show_engine_down_banner, _handle_engine_exit -->

**Симптом.** Постоянный небойкий (non-modal) баннер вверху окна и
повторяющийся звуковой сигнал, пока engine не поднят.

**Проверка.** Различить два текста баннера:

- **«ОШИБКА КОНФИГУРАЦИИ: Engine не запускается. Автоперезапуск
  отключён.»** — это случай №1 (exit code 2). Автоматических попыток не
  будет.
- **«Engine остановлен — перезапуск через N с (попытка K)»** — обычный
  крэш (код возврата ≠ 2). Лаунчер повторяет попытки бесконечно с
  экспоненциальной задержкой (кап 120 с) — «никогда не сдаётся», чтобы
  необслуживаемый прогон не умер тихо.

**Причина.** Симптом идентичен внешне (баннер + звук), но исправление
разное — не путайте один случай с другим по одному только факту
баннера.

**Исправление.** Для конфиг-ошибки — см. пункт 1. Для повторяющегося
крэша — открыть `logs/engine.log`, найти traceback ПЕРЕД первым
«Engine crashed» и устранить причину падения; бесконечный retry
маскирует проблему, а не решает её.

## 3. TSP watchdog Keithley: CRITICAL при активации

<!-- src/cryodaq/drivers/instruments/keithley_2604b.py: _wdog_arm, best_effort/required paths -->

**Симптом.** В логе `log.critical` со строкой про TSP watchdog при
подключении к Keithley 2604B.

**Проверка.** Три разных сообщения — разная причина:

- `SAFETY DEGRADED: watchdog is NON-AUTONOMOUS` — штатное честное сообщение
  для v3 в `best_effort`: работает только late-pet check, полной смерти хоста
  он не покрывает.
- `TSP watchdog upload/activation FAILED — running host-only` — не
  активировалась даже late-pet проверка; host-side контур — единственная защита.
- `TSP watchdog arm FAILED and mode=required — refusing to connect` —
  режим `required`: connect() отказал полностью, прибор недоступен.
- `TSP watchdog read back a LATCHED trip` — TSP помнит прошлое срабатывание
  на позднем pet;
  выходы уже принудительно выключены при коннекте, повторное
  взведение идёт штатно.

**Причина.** Для `required` текущая v3 намеренно не подходит:
`cryodaq_wdog_autonomous=0`. Для остальных сообщений причина — version stamp
mismatch, ошибка отдельного autonomous readback, сбой active/tripped readback
или latch от предыдущего late-pet срабатывания.

**Исправление.** Проверить `config/instruments.local.yaml` →
`keithley.watchdog.mode`. Для деградации в `best_effort` — переподключить
прибор (переподача TSP-скрипта происходит на каждый arm); если ошибка
повторяется — проверить GPIB/USB-линк и версию прошивки TSP. В
`required` — недоступность означает, что CryoDAQ не разрешит RUN, пока не
появится отдельно доказанный autonomous contract. Это не доказывает
физическое отсутствие энергии без независимого измерения. Полный стендовый
проход — `docs/lab_verification_checklist.md`, раздел 4.

Если лог сообщает unconsumed prior-trip evidence, не перезапускать цикл
автоматически. Выполнить emergency OFF, проверить причину, затем обычный
operator fault-ack с reason: только этот путь после повторного verified OFF
очищает latch и реактивирует late-pet check. При ошибке ack FAULT остаётся,
сообщение требует повторить OFF/ack или явный disconnect/reconnect. В
`required` v3 не может реактивироваться как autonomous: сначала изменить mode
на `best_effort` осознанно. `off` полностью отключает TSP-path и не является
успешным consume latch.

## 4. Сбой персистентности: locked-DB / disk-full

<!-- src/cryodaq/storage/sqlite_writer.py: _disk_full, _locked_failure_count, _signal_persistence_failure -->

**Симптом.** В логе `CRITICAL`: `DISK FULL detected in SQLite write` или
`LOCKED DB: N consecutive database is locked/busy failures spanning
Ms`. safety fault срабатывает через зарегистрированный persistence-failure
callback.

**Проверка.** Свободное место на диске — видно в нижней строке
состояния GUI (bottom status bar). Для locked-DB: сигнал срабатывает
только после **≥3 подряд** сбоев `database is locked`/`database is
busy`, растянутых на **≥15 секунд** — единичные транзиентные блокировки
(WAL-контеншн) не эскалируются, только устойчивая серия.

**Причина.** Disk-full — реально кончилось место на разделе с
`data/`. Locked-DB — как правило, посторонний читатель/писатель того же
SQLite-файла в обход `cryodaq.storage._sqlite` (прямой `import sqlite3`
на том же файле параллельно с движком).

**Исправление.** Disk-full — освободить место; флаг снимается
автоматически на первой успешной записи после восстановления места.
Locked-DB — найти и остановить конкурентный процесс/подключение,
которое держит файл вне штатного пути через `cryodaq.storage._sqlite`;
счётчик серии сам сбрасывается на первой успешной записи.

## 5. ZMQ command bridge уходит в restart-loop

<!-- src/cryodaq/core/zmq_bridge.py: _on_serve_task_done -->

**Симптом.** В логе повторяются строки `ZMQCommandServer serve loop
crashed; restarting` или `...exited unexpectedly; restarting`. GUI
может переставать отвечать на команды, при этом сбор данных
продолжается (REP-петля — только командный канал, не путь записи).

**Проверка.** Если следом идёт `ZMQCommandServer loop is closed; cannot
restart serve loop` — командный сервер умер окончательно (event loop
закрыт), самовосстановления не будет.

**Причина.** Необработанное исключение в serve loop либо неожиданный
выход корутины. Сервер сам перезапускает REP-петлю через
`call_soon`, но только пока event loop жив.

**Исправление.** Разовые перезапуски — не повод для действия (петля
сама восстанавливается). Если сообщение о закрытом loop — требуется
полный перезапуск процесса `cryodaq-engine` (кнопка «Перезапустить
Engine» или ручной рестарт).

## 6. Telegram молчит (тревога есть, сообщения нет)

<!-- src/cryodaq/notifications/telegram.py: _send_to_all/except paths; engine.py: _dispatch_alarm_notification fire-and-forget -->

**Симптом.** Алармы срабатывают (видны в GUI, в `Тревоги`), но
Telegram-уведомление не приходит, и GUI НЕ показывает никакой ошибки —
диспетчеризация в Telegram fire-and-forget по дизайну, сбой не
всплывает в интерфейс.

**Проверка.** Единственное место, где виден сбой — `logs/engine.log`:

```
Telegram API ответил <код>: <тело>
Ошибка отправки Telegram-уведомления: <exc>
```

**Причина.** Неверный/просроченный `bot_token`, неверный `chat_id`,
сетевая недоступность `api.telegram.org`, либо (dev-конфигурация)
`verify_ssl: false` за VPN/SSL-инспекцией.

**Исправление.** Проверить `config/notifications.yaml` (`bot_token`,
`chat_id`), проверить связность с `api.telegram.org` с лабораторного
ПК, свериться с логом на конкретный HTTP-код или exception. Отсутствие
записи об ошибке в логе при этом = сообщение реально ушло, ищите
проблему на стороне Telegram-бота/чата.

## 7. Датчик «протух» (stale), но тревога/fault не поднялись

<!-- src/cryodaq/core/safety_manager.py: on_interlock_dead_channel -->

**Симптом.** Канал явно не обновляется (видно в оверлее `Приборы` —
статус `Нет данных`/`Нет связи`, старое время последнего ответа), но
`SafetyManager` не переходит в `FAULT_LATCHED`.

**Проверка.** Текущая фаза эксперимента — если состояние **не**
`RUNNING` (idle, до старта, между экспериментами), устойчиво мёртвый
канал логируется как `CRITICAL`, но **никогда** не приводит к fault —
это осознанное поведение, не баг: вне `RUNNING` fault заблокировал бы
восстановление readiness без причины.

**Причина.** `SafetyManager` — единственный арбитр эскалации по
дедлайну RUNNING: только во время активного прогона мёртвый канал
(защищённый исключительно интерлоками, не критическими Т11/Т12)
латчит `FAULT_LATCHED` + `emergency_off`.

**Исправление.** Не ждите автоматической эскалации вне `RUNNING`.
Перед стартом эксперимента открывайте `Приборы` (Ctrl+D) и проверяйте
health-score каждого канала вручную — если датчик протух до старта,
это не будет поймано safety-контуром автоматически, только визуально.
